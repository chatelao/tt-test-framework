# Pattern-Guided Arithmetic Optimizations with MLIR per-op

**Source:** [https://github.com/Bynaryman/ttihp04_ieee754_perop](https://github.com/Bynaryman/ttihp04_ieee754_perop)

**TinyTapeout Project Page:** [https://app.tinytapeout.com/projects/4154](https://app.tinytapeout.com/projects/4154)

## Overview

This project implements a streaming floating-point operator generated via an MLIR-to-CIRCT toolchain. It demonstrates a compiler-visible path for floating-point arithmetic, lowering high-level arithmetic kernels into explicit hardware structures.

## How it Works

The design consists of a streaming wrapper (`tt_um_lledoux_s3fdp_seqcomb`) around a generated floating-point core. The core implements a simple loop-accumulation kernel:
```mlir
scf.for %k = %c0 to %c2 step %c1 {
  %x = memref.load %a[%k] : memref<2xf32>
  %y = memref.load %b[%k] : memref<2xf32>
  %sum_xy = arith.addf %x, %y : f32
  %prod = arith.mulf %sum_xy, %x : f32
  memref.store %prod, %c[%c0] : memref<2xf32>
}
```
The wrapper handles byte-serial loading of two 32-bit floating-point arrays (`a` and `b`, each with 2 elements) and a 32-bit constant `c0` (unused in the current kernel). After loading 20 bytes, it runs the computation for 3 cycles and then streams out the 32-bit result over 4 bytes.

## Register / Streaming Interface

The chip uses a simple streaming protocol on `ui_in` and `uo_out`.

### Protocol Timing
- **Load Phase**: 20 bytes are shifted in via `ui_in` (A[0], A[1], B[0], B[1], C0 - all little-endian).
- **Run Phase**: 3 clock cycles of latency.
- **Output Phase**: 4 bytes are shifted out via `uo_out` (little-endian).
- **Total Slot**: 27 cycles.
